I was blown away by the post [Breaking Up with SVG-in-JS in 2023](https://kurtextrem.de/posts/svg-in-js), and I bet it prompted many of us to start rethinking how we're using SVG in our work.

We use it primarily for icons, often through npm packages, so that's what I'm going to talk about in this post. Occasionally much bigger SVGs find their way into our bundle as well, but either way we should start switching to external SVG files.

I'd like to examine the two most common approaches for using SVG icons — SVG-in-JS and SVG sprites.

**Note**: if SVG sprites are unfamiliar to you (`<symbol>{:xml}` + `<use>{:xml}`), the post [The "best" way to manage icons in React.js](https://benadam.me/thoughts/react-svg-sprites/) will quickly bring you up to speed.

## Icons with SVG-in-JS

I'd like to start by saying that I wasn't even fully aware that I had been using using SVG-in-JS! I assume it's because of the following reasons:

1. JSX doesn't look like repeating calls to `React.createElement`, so using SVG there feels like regular inline SVG to me
2. `<path>{:html}` elements can be hell of a lot bigger than they initially seem
3. using icons through packages hides their SVG code

To use an open source set of icons like [Radix Icons](https://www.radix-ui.com/icons) or [Heroicons](https://heroicons.com/) we can usually install it by running something like `npm install @radix-ui/react-icons` and import the ones we want to use:

```jsx
import { FaceIcon, ImageIcon, SunIcon } from '@radix-ui/react-icons'

function MyComponent () {
  return (
    <div>
      <FaceIcon />
      <SunIcon />
      <ImageIcon />
    </div>
  )
}
```

### Downsides

  - icon's JSX code takes up at least 3x more file size than its SVG equivalent
  - a lot of inline SVG means increases the size of the HTML document, which hurts runtime performance and memory usage
  - pasting any custom SVG code into JSX needs attribute adjustments, for example `fill-rule` needs to become `fillRule`, which is very tedious
    - fortunately, this can be autofixed with [`react/no-unknown-property`](https://github.com/jsx-eslint/eslint-plugin-react/blob/d73cd516aa21e2331e321e4cd28258347622221b/docs/rules/no-unknown-property.md)

### Benefits

	- easy to use, just `npm install` + `import`
	- our bundle contains only the icons that we use
  - editing SVG code in JSX is easier (IntelliSense, Prettier, ESLint, TypeScript...)

For each icon's SVG I had to make quite a lot of adjustments:

1. changed `<svg>{:xml}` to `<symbol>{:xml}`
2. removed the following attribues from `<symbol>{:xml}`: `xmlns`, `width`, `height`, `fill`, `stroke`
3. removed the following attributes from `<path>{:xml}`: `fill`, `stroke`
4. added an `id` to `<symbol>{:xml}`, which serves as the icon name

Attributes that represent size and color need to be removed because I might want those to be customizable upon usage:

```jsx
function Icon({ icon, ...svgProps }) {
  return (
    <svg {...svgProps}>
      <use href={`/sprite.svg#${icon}`} />
    </svg>
  )
}

function MyComponent () {
  return (
    <div>
      <Icon icon="face" width="15" height="15" fill="currentColor" />
      <Icon icon="sun" width="15" height="15" fill="currentColor" />
      <Icon icon="image" width="15" height="15" fill="currentColor" />
    </div>
  )
}
```

### Downsides

  - adjustments are needed for adding icons to the sprite
  - no IntellSense, SVG is just XML to VS Code
  - no Prettier either, but sometimes I trick it by parsing SVG as HTML using <span className="whitespace-nowrap">`--parser=html`</span>
  - resources about sprites are outdated, so I recommend bookmarking the catalyst article [Breaking Up with SVG-in-JS in 2023](https://kurtextrem.de/posts/svg-in-js) as a reference

### Benefits

  - smaller bundle size
  - faster runtime performance
  - smaller `node_modules` size
  - when you get the hang of it, you won't feel limited to your installed icon set
    - check out [Iconify](https://icon-sets.iconify.design/) to have some fun!

## Workflows? Tools?

If done manually, managing an SVG sprite can be tricky, adding new icons is much slower due to repetitive adjustments, also, unused icons remain in the sprite... it would be nice to have some tools to help us.

### My ideal workflow

When adding a new icon to my project I'm either in Figma or browsing an open source icon set's website. In either case it's usually easy for me to copy the icon's SVG code, so the workflow I'm actually looking for is:

  1. paste the icon's SVG code somewhere as-is
  2. being able to import the icon as a React component
  3. a git-ignored SVG sprite gets created containing only imported icons

The "tree-shaking" aspect of this workflow sounds impossible without a bundler plugin, but unused icons ending up in the sprite isn't the worst thing ever, so that workflow can be achieved with a tool or a custom script.

### Existing tools

[Sly CLI](https://sly-cli.fly.dev//cli) is a tool for adding components from supported registries, including icons. Among other things it supports adding custom transformer scripts to automate repetitive adjustments, like what we're doing to add icons to our sprite. So a Radix UI icon file added by Sly CLI can look like this:

Sly CLI also supports defining a script that will run every time new components are added, that's a good place to run a script for bundling added icons into a sprite.

Epic Stack for Remix uses this workflow, check out its [icons documentation](https://github.com/epicweb-dev/epic-stack/blob/6da99116c26b82d846476b15363c642a4551428c/docs/icons.md). Some of the logic there is explained by Sly CLI's creator, Jacob Paris, in [Use SVG sprite icons in React](https://www.jacobparis.com/content/svg-icons).

However, considering that I can paste icon's SVG code into a new SVG file, I currently don't see the need for Sly CLI in my projects, it would add a layer of indirection. I will probably combine the transformer + bundling logic into the same script.

## Conclusion

Manually handling an SVG sprite is fine for a small project, but in a team handling icons needs to be more robust to minimize mistakes. Until the ecosystem catches up we'll need to write custom scripts or copy the ones from the Epic Stack.
